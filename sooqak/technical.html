🛠 البنية التقنية الحالية

سوقك مبني على 6amMart من CodeCanyon. هذه هي التقنيات الأساسية:

L

Laravel (PHP)

الباك إند بالكامل — API، لوحات التحكم (Blade)، قاعدة البيانات (MySQL)، والمصادقة.

  • REST API للتطبيقات
  • Blade Templates للوحات التحكم
  • Eloquent ORM لقاعدة البيانات
  • Laravel Middleware للصلاحيات
F

Flutter (Dart)

تطبيقات الموبايل الثلاثة — العميل، التاجر، وسائق التوصيل. كلها Flutter.

  • تطبيق العميل (Customer App)
  • تطبيق التاجر (Store App)
  • تطبيق التوصيل (Delivery App)
  • GetX لإدارة الحالة
M

MySQL

قاعدة البيانات الرئيسية. تحتاج تعديلات جوهرية لدعم نظام الوكلاء.

  • جداول جديدة للدول والوكلاء
  • تعديل العلاقات الموجودة
  • إضافة حقول العملة لكل دولة
  • فهارس للأداء

🗃 تعديلات قاعدة البيانات

هذه أهم التعديلات المطلوبة على مستوى قاعدة البيانات:

جداول جديدة مطلوبة:

جديد countries

جدول الدول — يحتوي على:

  • اسم الدولة (عربي + إنجليزي)
  • كود الدولة (ISO)
  • العملة ورمزها
  • نسبة الأدمن من إيرادات هذه الدولة
  • الحالة (مفعّلة / معطّلة)
  • بوابات الدفع المتاحة

جديد agents

جدول الوكلاء — يحتوي على:

  • بيانات الوكيل (مرتبط بجدول users)
  • الدولة المسؤول عنها (country_id)
  • بيانات بوابة الدفع الخاصة به
  • إعدادات التوصيل في دولته
  • نسبته من الإيرادات

جديد zone_managers

جدول مديري المناطق — يحتوي على:

  • بيانات المدير (مرتبط بجدول users)
  • المنطقة المسؤول عنها (zone_id)
  • نسبته من حصة الوكيل

جديد agent_payment_gateways

بوابات الدفع لكل وكيل:

  • نوع البوابة (Fawry, PayMob, Stripe...)
  • بيانات الربط (API keys, secrets)
  • مرتبطة بالوكيل (agent_id)
  • الحالة (مفعّلة / معطّلة)

جديد monthly_invoices

الفواتير الشهرية التلقائية:

  • الدولة والوكيل (country_id, agent_id)
  • الشهر والسنة
  • إجمالي الإيرادات
  • نسبة ومبلغ الأدمن المطلوب
  • الحالة (معلّقة / مدفوعة / متأخرة)
  • تاريخ الدفع

جداول تحتاج تعديل:

الجدول التعديل المطلوب السبب
zones إضافة country_id + zone_manager_id + commission_rate ربط الزون بالدولة ومدير المنطقة ونسبته
stores إضافة country_id + حقول التخصيص (colors, description, contact_info) ربط المتجر بالدولة وتمكين التخصيص الكامل
users إضافة role (agent, zone_manager) + country_id دعم الأدوار الجديدة وربطها بالدول
orders إضافة country_id + currency تتبع الطلبات حسب الدولة والعملة
delivery_men إضافة country_id + zone_id + approved_by (zone_manager) + is_store_driver + store_id ربط السائق بالمنطقة ومديرها، ودعم سائقي التاجر الخاصين
coupons إضافة country_id + created_by_agent الكوبونات تُنشأ من الوكيل لدولته
subscription_packages إضافة حقول الميزات (has_pos, has_chat, max_products, max_orders...) دعم الباقات المتعددة المستويات

⚙ تعديلات الباك إند (Laravel)

1

نظام الصلاحيات المتعدد المستويات

إنشاء Middleware جديد يفحص دور المستخدم (admin, agent, zone_manager, store) ويحدد البيانات التي يمكنه الوصول إليها. كل دور يرى فقط البيانات المرتبطة بنطاقه (دولته أو منطقته).

2

API Scoping حسب الدولة

كل API endpoint يجب أن يُفلتر حسب الدولة. العميل يرى متاجر دولته فقط. الوكيل يدير بيانات دولته فقط. هذا يتطلب Global Scope أو Middleware على مستوى الـ Query.

3

لوحة الوكيل (جديدة)

بناء لوحة تحكم كاملة للوكيل بـ Laravel Blade. تشمل: إدارة المناطق، مديري المناطق، التوصيل، بوابات الدفع، الكوبونات، الدعم الفني، والتقارير المالية.

4

لوحة مدير المنطقة (جديدة)

لوحة مبسطة لمدير المنطقة: إدارة التجار في منطقته، التقارير المالية الخاصة بمنطقته فقط.

5

نظام بوابات الدفع المتعددة

تعديل Payment Gateway logic بحيث تُحمّل بيانات البوابة من جدول الوكيل وليس من إعدادات الأدمن. كل وكيل يربط بواباته الخاصة.

6

نظام العملات

إضافة دعم العملات المتعددة: عرض الأسعار بعملة الدولة، حساب الإيرادات والتقارير بالعملة المحلية.

7

نظام توزيع الأرباح

بناء Service لحساب توزيع الأرباح تلقائياً: حصة الأدمن، حصة الوكيل، حصة مدير المنطقة. مع تقارير شهرية لكل طرف.

8

تعديل لوحة الأدمن

إضافة إدارة الدول والوكلاء. إزالة صلاحيات التوصيل والدعم والكوبونات من الأدمن (تنتقل للوكيل).

9

تعديل لوحة التاجر

إضافة خيارات تخصيص صفحة المتجر (ألوان، صور، وصف، بيانات تواصل). تطبيق قيود الباقات (حد المنتجات، حد الطلبات، الميزات المتاحة).

📱 تعديلات الفرونت إند (Flutter)

تطبيق العميل

  • إضافة شاشة اختيار/تحديد الدولة عند أول استخدام
  • فلترة كل البيانات (متاجر، منتجات، عروض) حسب الدولة
  • عرض الأسعار بعملة الدولة المحددة
  • استخدام بوابة الدفع الخاصة بالدولة (بيانات الوكيل)
  • توجيه الشات مع خدمة العملاء إلى الوكيل بدلاً من الأدمن

تطبيق التاجر

  • إضافة واجهة تخصيص صفحة المتجر
  • تطبيق قيود الباقة (حد المنتجات، إخفاء ميزات غير متاحة)
  • عرض العملة المحلية حسب الدولة

تطبيق التوصيل

  • تعديل عملية التسجيل — طلب القبول يروح لمدير المنطقة مش الأدمن
  • عرض العملة المحلية
  • التوصيل محصور في نطاق المنطقة (الزون)
  • دعم سائقي التاجر الخاصين (نفس التطبيق، تكلفة مختلفة)

🔒 اعتبارات أمنية

🔐 عزل البيانات بين الدول

كل وكيل يجب أن يرى فقط بيانات دولته. لا يمكن لوكيل مصر الوصول لبيانات السعودية. هذا يتطلب فلترة صارمة على مستوى الـ Query وليس فقط الـ UI.

🔑 حماية بيانات بوابات الدفع

بيانات بوابات الدفع (API keys, secrets) حساسة جداً. يجب تشفيرها في قاعدة البيانات وعدم عرضها في الـ API responses.

🛡 التحقق من الصلاحيات

كل endpoint يجب أن يتحقق من: هل هذا المستخدم له الحق في هذا الإجراء؟ وهل البيانات المطلوبة تقع في نطاقه (دولته/منطقته)؟

💰 حماية العمليات المالية

عمليات الدفع وتوزيع الأرباح يجب أن تكون atomic (معاملات قاعدة بيانات). أي خطأ يجب أن يُرجع كل شيء لحالته الأصلية.

🌐 اللغات والتعريب

المرحلة الأولى

  • العربية (RTL) — اللغة الأساسية
  • الإنجليزية (LTR) — اللغة الثانوية

6amMart يدعم أصلاً تعدد اللغات عبر ملفات الترجمة. المطلوب: ترجمة كل النصوص الجديدة (الوكيل، مدير المنطقة، إلخ).

المراحل اللاحقة

  • إضافة لغة كل دولة جديدة (تركي، أوردو، فرنسي...)
  • الوكيل يحدد اللغات المتاحة في دولته

البنية موجودة — فقط إضافة ملفات ترجمة جديدة لكل لغة.
