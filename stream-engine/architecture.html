البنية والمكونات

المكونات الرئيسية في Stream Engine وكيف تعمل معاً لتقديم منصة اتصالات متكاملة

🛠 المكونات السبع

كل مكوّن له دور محدد ويعمل بشكل مستقل — لكنها جميعاً تتكامل معاً لتشكيل النظام الكامل.

🌐

طبقة الـ API

نقطة الدخول لكل شيء — REST APIs يستخدمها سيرفر المطور للتحكم في كل الخدمات.

  • APIs مخصصة للمطورين (Client API) — إنشاء غرف، توليد Tokens، إدارة مكالمات
  • APIs للإدارة الداخلية (Admin API) — مراقبة، إحصائيات، تعديل إعدادات
  • توثيق موحد بـ App ID + App Secret
  • Rate limiting وحماية ضد الاستخدام المفرط
🎥

محرك الميديا — LiveKit

القلب النابض — مسؤول عن نقل الصوت والفيديو بين المستخدمين في الوقت الحقيقي.

  • مبني على LiveKit — مشروع مفتوح المصدر رائد في مجال WebRTC
  • يدعم SFU (Selective Forwarding Unit) لأداء عالي
  • منتشر في أكثر من منطقة جغرافية لتقليل التأخير
  • يدعم 5 جودات فيديو: SD, HD, FHD, 2K, 2K+
📋

نظام الخدمات الخمس

خمس خدمات مستقلة تماماً — كل واحدة بقواعدها وإعداداتها الخاصة.

  • غرف صوتية — صوت فقط، أدوار، بدون فيديو أبداً
  • بث مباشر — صوت + فيديو + شاشة، تفاعلي
  • مكالمات — 1-to-1، صوت أو فيديو، signaling مدمج
  • دردشة فورية — نص وملفات، فردية ومجموعات
  • نظام حضور — تتبع متصل/غير متصل
🔒

نظام التوثيق والأمان

طبقة أمان شاملة تحمي كل مشروع وكل مستخدم.

  • App ID + App Secret لكل مشروع
  • Tokens محدودة الصلاحية والوقت لكل مستخدم
  • عزل كامل بين المشاريع (Tenant Isolation)
  • Token يحمل الخدمة والدور والجودة القصوى
⚡

نظام الاتصال الفوري — WebSocket

اتصال مستمر بين تطبيق المستخدم والسيرفر — لكل ما هو غير ميديا.

  • الدردشة الفورية — إرسال واستقبال الرسائل لحظياً
  • إشعارات المكالمات — رنين، قبول، رفض، مشغول، إلغاء
  • تتبع الحضور — من متصل ومن غير متصل
  • اتصال مستمر وإعادة اتصال تلقائية
📈

نظام تتبع الاستخدام

يسجل كل دقيقة صوت وفيديو لكل مشروع — بدقة عالية.

  • تتبع بالدقيقة لكل مشارك في كل غرفة
  • تفصيل جودة الفيديو الفعلية (SD, HD, FHD, 2K, 2K+)
  • فصل الاستهلاك حسب الخدمة (غرف، بث، مكالمات)
  • ملخص يومي تلقائي يُرسل لسيرفر المطور
🔔

نظام الأحداث — Webhooks

يُبلّغ سيرفر المطور فوراً عند حدوث أي شيء في النظام.

  • أحداث فورية: مشارك دخل، غرفة انتهت، مكالمة فائتة
  • موقّع رقمياً للأمان (HMAC-SHA256)
  • إعادة المحاولة تلقائياً عند فشل التسليم
  • سجل كامل لكل الأحداث المُرسلة

⚖ ما يقدمه LiveKit وما يضيفه UTD

LiveKit مشروع مفتوح المصدر قوي — لكنه مجرد محرك ميديا. UTD Stream Engine يبني فوقه نظاماً متكاملاً.

الجانب LiveKit يقدم UTD يضيف فوقه
الخدمات غرف بدون تصنيف — كلها نفس النوع 5 خدمات منفصلة بقواعد مختلفة لكل واحدة
النشر أي مشارك ينشر أي شيء (صوت، فيديو، شاشة) تحكم Server-side — الـ Token يحدد ما يُسمح به
المكالمات لا يوجد نظام مكالمات نظام مكالمات كامل مع Signaling (رنين، قبول، رفض)
الدردشة لا يوجد نظام دردشة نظام رسائل فوري متكامل — 7 أنواع رسائل
الحضور لا يوجد نظام حضور تتبع حضور دقيق — متصل/غير متصل/آخر ظهور
الاستخدام لا يوجد تتبع استخدام تتبع دقيق بالدقيقة حسب جودة الفيديو الفعلية
العزل لا يوجد عزل بين العملاء عزل كامل بين المشاريع (Tenant Isolation)
الأحداث لا يوجد نظام أحداث للمطور نظام Webhooks موقّع رقمياً (HMAC-SHA256)
الأدوار لا يوجد نظام أدوار 4 أدوار بصلاحيات مختلفة (host, guest, audience, visitor)
الـ Token Token بسيط — يسمح بالدخول فقط Token ذكي يحمل الخدمة والدور والجودة القصوى

🛡 الفرق بين Server-side و Client-side Enforcement

لماذا يعتبر فرض القيود من السيرفر أهم قرار أمني في تصميم Stream Engine.

Server-side Enforcement — القيود تُفرض من السيرفر لا من التطبيق.

في معظم المنصات، القيود تعتمد على تطبيق العميل — إذا عدّل المطور الكود أو استخدم أدوات تصحيح، يمكنه تجاوز القيود. في Stream Engine، الأمر مختلف تماماً:

مثال عملي: Token لخدمة "rooms" لا يسمح بنشر فيديو أبداً — حتى لو حاول المطور تفعيله من تطبيقه. السيرفر (LiveKit + UTD Layer) يرفض الطلب لأن الـ Token لا يحمل صلاحية الفيديو. نفس الشيء مع الأدوار: مستخدم بدور "audience" لا يمكنه نشر أي شيء مهما فعل في تطبيقه.

النتيجة: أمان حقيقي لا يعتمد على سلوك التطبيق — يعتمد على السيرفر فقط.
❌

Client-side (غير آمن)

التطبيق يقرر ما يُسمح به. المطور يمكنه تجاوز القيود بتعديل الكود. أي ثغرة في التطبيق = ثغرة في النظام.

✅

Server-side (آمن)

السيرفر يقرر ما يُسمح به بناءً على الـ Token. لا يمكن تجاوزه من التطبيق. حتى لو اختُرق التطبيق، السيرفر يحمي النظام.
