التوثيق والأمان

نظام حماية ثلاثي الطبقات — من تعريف المشروع إلى عزل المستخدمين

🔒 النظام ذو ثلاث طبقات

تعريف المشروع (Project Auth)

كل مشروع يحصل على App ID (رقم فريد من 10 خانات) + App Secret (مفتاح سري). يُرسلان مع كل طلب API في Headers. Engine يتحقق أن المشروع والعميل موجودان ونشطان. هذه المفاتيح تبقى في سيرفر المطور فقط — لا تُكشف للمستخدم النهائي أبداً.

توكن الميديا (Media Token)

سيرفر المطور يطلب token من Engine عبر API. Engine يُولّد LiveKit Access Token يحمل: هوية المستخدم، اسم الغرفة، الخدمة، الصلاحيات، الجودة القصوى. تطبيق المستخدم يستخدم هذا Token للاتصال بسيرفر الميديا مباشرة. مدة الصلاحية: ساعة واحدة. Token ذكي: يحمل قيود الخدمة — مثلاً token لخدمة "rooms" لا يسمح بنشر فيديو.

توكن الـ Signaling (Signal Token)

سيرفر المطور يطلب Signal Token من Engine. موقّع رقمياً (HMAC-SHA256). تطبيق المستخدم يتصل عبره بالـ WebSocket (للدردشة والحضور وإشعارات المكالمات). مدة الصلاحية: ساعة واحدة.

🔄 تدفق التوثيق

1. سيرفر المطور → Engine API

يرسل App ID + App Secret ← يطلب Tokens (Media Token أو Signal Token أو كلاهما)

2. Engine يُولّد الـ Tokens

يتحقق من المشروع والعميل ← يُولّد tokens محدودة الصلاحية والوقت

3. Media Token → LiveKit

تطبيق المستخدم يستخدم Media Token للاتصال بسيرفر LiveKit ← صوت وفيديو

4. Signal Token → WebSocket

تطبيق المستخدم يستخدم Signal Token للاتصال بالـ WebSocket ← دردشة وحضور وإشعارات مكالمات

🛡 عزل العملاء (Tenant Isolation)

عزل على مستوى API

كل طلب يتطلب App ID + App Secret — لا يمكن الوصول لبيانات مشروع آخر.

عزل على مستوى الغرف

أسماء الغرف في LiveKit تُنسب بمعرّف المشروع تلقائياً (Room Name Prefixing) — غرفة "meeting1" في مشروع A معزولة تماماً عن "meeting1" في مشروع B.

عزل على مستوى الإعدادات

كل مشروع له إعداداته المستقلة: الخدمات المفعلة، الحدود، الجودة، الـ callbacks.

عزل على مستوى البيانات

بيانات الاستخدام مفصولة بالكامل لكل مشروع — لا تختلط أبداً.
