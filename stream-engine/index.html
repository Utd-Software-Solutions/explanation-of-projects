⚡ UTD Stream Engine

البنية التحتية السحابية (CPaaS) التي تشغّل خدمة UTD Stream. منصة اتصالات سحابية تقدم خدمات الصوت والفيديو والدردشة عبر APIs جاهزة للمطورين — بدلاً من بناء بنية تحتية معقدة من الصفر.

4 خدمات مستقلة
< 300ms تأخير
Server-side Enforcement
5 جودات فيديو

💡 المشكلة والحل

لماذا يحتاج المطور إلى Stream Engine بدلاً من بناء كل شيء بنفسه؟

❌

المشكلة

المطور الذي يريد إضافة مكالمات فيديو أو دردشة صوتية أو رسائل فورية لتطبيقه يحتاج بنية تحتية معقدة:

  • سيرفرات ميديا متخصصة
  • بروتوكولات WebRTC وإعدادات TURN/STUN
  • نظام Signaling كامل
  • تتبع استخدام وفوترة
  • أمان وعزل بين المستخدمين

بناء كل ذلك من الصفر يستغرق أشهراً ويكلف ملايين.

✅

الحل — Stream Engine

بدلاً من بناء كل ذلك، المطور يستخدم APIs من Stream Engine ويحصل على كل شيء جاهز:

  • غرف صوتية جاهزة بأدوار وصلاحيات
  • بث مباشر تفاعلي بتأخير أقل من 300ms
  • مكالمات صوتية ومرئية بنظام signaling مدمج
  • دردشة فورية بـ 7 أنواع رسائل
  • تتبع استخدام دقيق بالدقيقة

يبدأ البناء في ساعات بدلاً من أشهر.

🔧 الخدمات الأربع

كل خدمة مستقلة تماماً — لها إعداداتها وصلاحياتها وتتبع استخدامها الخاص.

🎙

غرف الدردشة الصوتية

صوت فقط — بدون فيديو. أدوار مختلفة (host، guest، audience). الخدمة الأكثر شيوعاً في تطبيقات منطقة MENA.

  • صوت فقط — بدون فيديو أبداً
  • عدد غير محدود من المشاركين
  • أدوار وصلاحيات مفروضة من السيرفر
  • مناسبة لتطبيقات مثل Clubhouse والبودكاست المباشر
📺

البث المباشر

صوت + فيديو + مشاركة شاشة. بث تفاعلي بتأخير أقل من 300ms مع دعم Co-hosting.

  • صوت وفيديو ومشاركة شاشة
  • بث تفاعلي وليس بث أحادي الاتجاه
  • تأخير أقل من 300ms
  • دعم Co-hosting — أكثر من مضيف في نفس الوقت
📞

المكالمات الصوتية والمرئية

مكالمات 1-to-1 — صوت أو فيديو. نظام Signaling جاهز (رنين، قبول، رفض، مشغول).

  • مكالمات بين شخصين فقط
  • صوت فقط أو صوت + فيديو
  • نظام Signaling مدمج وجاهز
  • مناسبة لتطبيقات المراسلة والاستشارات
💬

الدردشة الفورية

محادثات فردية ومجموعات — 7 أنواع رسائل مختلفة عبر WebSocket.

  • محادثات فردية ومجموعات
  • 7 أنواع رسائل (نص، صورة، صوت، فيديو، ملف، موقع، رد)
  • إشعارات فورية عبر WebSocket
  • حالات الرسائل (مُرسلة، مُستلمة، مقروءة)

🔗 العلاقة مع UTD Stream

UTD Stream و UTD Stream Engine وجهان لعملة واحدة — أحدهما تجاري والآخر تقني.

UTD Stream = الواجهة التجارية
الموقع، لوحة التحكم، التسعير، المحفظة، الباقات، الفواتير، الدعم. هذا ما يتعامل معه العميل (صاحب التطبيق) يومياً — يسجل، يدفع، يراقب استهلاكه، يدير مشاريعه.
UTD Stream Engine = الواجهة التقنية
APIs، Tokens، غرف، مكالمات، دردشة، Webhooks، WebSocket. هذا ما يتعامل معه مطور العميل — يبني التطبيق، يدمج الخدمات، يستقبل الأحداث.
باختصار: المطور يسجل ويدفع عبر UTD Stream (الموقع والداشبورد)، ويبني تطبيقه عبر UTD Stream Engine APIs (الكود والتكامل). المنصتان تعملان معاً كنظام واحد متكامل.

🚀 مخطط التدفق

كيف يتدفق الاتصال من سيرفر المطور إلى المستخدم النهائي — خطوة بخطوة.

1

سيرفر المطور ← Stream Engine API

سيرفر المطور يرسل طلب إلى Stream Engine API مع App ID + App Secret لإنشاء غرفة أو بدء مكالمة أو فتح محادثة. هذا الطلب يتم من السيرفر فقط — لا من تطبيق المستخدم مباشرة.

2

Stream Engine API ← توليد Tokens

Stream Engine يولّد Tokens ذكية لكل مستخدم. كل Token يحمل: الخدمة المسموحة، الدور (host/guest/audience)، الجودة القصوى للفيديو، ومدة الصلاحية. المطور يرسل هذه الـ Tokens لتطبيق المستخدم.

3

تطبيق المستخدم ← Media Token → LiveKit Server

لخدمات الصوت والفيديو (غرف، بث، مكالمات): تطبيق المستخدم يستخدم Media Token للاتصال بـ LiveKit Server مباشرة. LiveKit يتولى نقل الصوت والفيديو بين المشاركين في الوقت الحقيقي عبر WebRTC.

4

تطبيق المستخدم ← Signal Token → WebSocket Server

لخدمات الدردشة والحضور وإشعارات المكالمات: تطبيق المستخدم يستخدم Signal Token للاتصال بـ WebSocket Server. هذا الاتصال المستمر يُستخدم لإرسال واستقبال الرسائل الفورية، إشعارات الرنين والقبول والرفض، وتحديثات الحضور.

5

الأحداث → سيرفر المطور (Webhooks)

كل ما يحدث (مشارك دخل غرفة، مكالمة فائتة، رسالة أُرسلت) — Stream Engine يُبلّغ سيرفر المطور فوراً عبر Webhooks موقّعة رقمياً. المطور يستخدم هذه الأحداث لتحديث قاعدة بياناته وتطبيقه.
