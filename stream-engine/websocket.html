بروتوكول WebSocket

صيغة الرسائل، أنواع الطلبات، أكواد الأخطاء — للمطور المتقدم

🔌 نقطة الاتصال

عنوان الاتصال

wss://engine.udt-stream.com/ws?token=TOKEN

المصادقة: Signal Token يُمرر كـ query parameter عند فتح الاتصال. التطبيق يحصل على Signal Token من سيرفر المطور (الذي يطلبه من Engine API).

أو عبر UTD ZIM SDK

المطور يمكنه استخدام UTD ZIM SDK (utd_zim) بدلاً من التعامل مع WebSocket مباشرة — الـ SDK يتولى كل تفاصيل الاتصال والـ reconnection تلقائياً.

📄 صيغة الرسائل

طلب من العميل

كل طلب يحمل: type (نوع الطلب)، id (معرّف فريد لتتبع الرد)، وdata (البيانات المطلوبة). مثال: type: "message.send"، id: "req_123"، data: محتوى الرسالة.

تأكيد (Acknowledgment)

السيرفر يرد بـ: type: "ack"، id (نفس معرّف الطلب)، code (كود النتيجة — 0 = نجاح)، وdata (البيانات المُرجعة إن وُجدت).

إشعار من السيرفر (Push)

السيرفر يرسل إشعارات بدون طلب مسبق: type (اسم الحدث — مثل "callInvitation" أو "messageReceived")، وdata (بيانات الحدث).

💓 Heartbeat (نبض الاتصال)

كيف يعمل

السيرفر يرسل ping كل 30 ثانية. العميل يجب أن يرد بـ pong خلال 10 ثوانٍ. إذا لم يستجب العميل، يُقطع الاتصال تلقائياً.

لماذا مهم

يضمن أن الاتصال حي وأن المستخدم لا يزال متصلاً. عند انقطاع الاتصال، يتم تحديث حالة الحضور وتنظيف الاشتراكات تلقائياً.

🛡 حدود الاستخدام (Rate Limiting)

الحد لكل اتصال

60 رسالة في الثانية لكل اتصال WebSocket. إذا تجاوز العميل هذا الحد، يحصل على كود خطأ 6 (تجاوز حد المعدل).

إعادة الاتصال التلقائي

UTD ZIM SDK يدعم إعادة الاتصال التلقائي عند انقطاع الشبكة — مع الحفاظ على الاشتراكات والحالة. المطور الذي يتصل مباشرة يحتاج بناء هذا بنفسه.
